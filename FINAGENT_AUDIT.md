## BUGS & RISKS

1. **Critical: reranked relevance is discarded and BM25-only scores escape the [0,1] contract.** `rag/vector_store.py:399-424` correctly applies sigmoid to cross-encoder logits, but `rag/vector_store.py:456-466` publishes raw `dense_score` or raw BM25 score multiplied by 0.8 instead. BM25 is unbounded, so `RetrievedPassage.similarity_score` can exceed 1 and is used by confidence thresholds in `agents/analysis.py:215-230`; a score-range mismatch can inflate the evidence adjustment and show impossible UI percentages. **Fix:** publish `rerank_confidence` when present; otherwise explicitly normalize a single bounded fusion score, never raw BM25.

2. **High: dense score conversion is mathematically inconsistent with the invoked FAISS API.** `rag/vector_store.py:252-256` calls `similarity_search_with_relevance_scores`, documented by its name/API as relevance-normalized, then transforms it again with `1 - score / 2`. This reverses/distorts its semantics (and compresses results to 0.5-1), while the comments claim a cosine conversion. **Fix:** use the returned relevance score directly after verifying the configured LangChain distance strategy, or use raw distances and one documented conversion.

3. **High: asset filter cannot guarantee company isolation.** `rag/vector_store.py:328-344` filters only whether an alias appears in `doc_name` or section title. Chunks have no canonical `asset` metadata (`rag/ingest.py:405-413`); a generic filename yields no results, and a section mentioning a competitor can pass for the wrong issuer. This fails the requested guarantee that one company's query cannot retrieve another's source. **Fix:** require/derive a canonical asset identifier at ingestion and filter exact metadata on it, with a documented policy for multi-company documents.

4. **High: document signal uploads are not ingested into the knowledge base.** `backend/routers/signals.py:122-138` calls `load_and_chunk_document`, takes only the first chunk excerpt, and runs the graph; it never calls `add_documents` or `save`. A submitted filing therefore cannot serve as future RAG context. **Fix:** index and persist all extracted chunks before/while processing the document event, returning count and embedding status.

5. **High: the document-submission UI sends a fabricated zero-valued price tick.** `frontend/src/pages/SignalSubmission.tsx:105-120` ignores the selected file and calls the price tick mutation with zero OHLCV. The backend receives neither a document nor its type and produces a malformed price event. **Fix:** add a document upload service that posts multipart data to `/signals/document` and call it here.

6. **High: backend/frontend data contracts break required UI features.** Backend decisions expose `artefact_id`, `created_at`, `confidence_score`, `alert_triggered`, and `evidence_bullets` (`core/state.py:211-225`), while frontend types/components expect `decision_id`, `timestamp`, `confidence`, `alert`, and `evidence` (`frontend/src/types/api.ts:173-190`; `frontend/src/pages/Dashboard.tsx:29-32`, `135-139`, `183-209`; `frontend/src/pages/LiveAnalysis.tsx:249-256`). This causes timestamps, alerts, confidence/history keys, and summaries to be blank or incorrect. **Fix:** define one shared API DTO and map server fields consistently, then add contract tests.

7. **High: trace and retrieval observability is lost when a decision is persisted.** `_persist_decision` writes only `artefact.model_dump_json()` (`backend/routers/signals.py:25-36`), while Trace Viewer loads individual persisted decisions and expects `trace_log` (`frontend/src/pages/TraceViewer.tsx:42-47`). It will always find no trace for API-generated history. **Fix:** persist an event record containing artefact, trace, errors, hypothesis, and attributed passages, or provide a durable trace endpoint keyed by event/artefact ID.

8. **Medium: retrieval errors and empty indexes are hidden as successful RAG.** `agents/retrieval.py:208-233` catches every vector error, inserts hard-coded generic passages (`112-182`), and emits status `ok` with no error. Analysis then treats them as supporting evidence. **Fix:** return an explicit fallback/error trace and state error; distinguish ungrounded framework advice from retrieved KB evidence and gate confidence accordingly.

9. **Medium: news summary and full text are silently discarded from model context.** `agents/ingestion.py:159-175` stores those fields yet derives `normalized_text` from only headline; `_build_query` and analysis consume `normalized_text` (`agents/retrieval.py:87-98`; `agents/analysis.py:159-172`). Material context can never affect retrieval or classification. **Fix:** normalize headline, summary, and full text with bounded length and provenance.

10. **Medium: ingestion silently converts missing/malformed OHLCV to zero and loses source timestamps.** `agents/ingestion.py:121-152` supplies `0.0` for absent OHLCV and uses `received_at` as the event time (`191-201`); `_save_price_history` suppresses all failures (`48-55`). Invalid price signals can appear valid, contaminate RSI/SMA, and have an incorrect timestamp. **Fix:** validate required finite OHLCV and optional event timestamp at the API boundary; surface persistence failures rather than silently continuing.

11. **Medium: file-backed price history and decision persistence are race-prone.** Price history is unsynchronized read-modify-write (`agents/ingestion.py:37-55`, `132-138`), and decision history is the same (`backend/routers/signals.py:25-36`). Concurrent FastAPI requests can overwrite entries or produce malformed JSON. **Fix:** move mutable state to a transactional store or protect file operations with a process-safe lock and atomic replace.

12. **Medium: decision history is oldest-first, contrary to reverse-chronological requirement.** Decisions are appended (`backend/routers/signals.py:35`) and listed from index zero (`backend/routers/decisions.py:23-27`) with no sort. The dashboard treats the first row as latest (`frontend/src/pages/Dashboard.tsx:29`). **Fix:** sort by `created_at` descending before pagination or prepend records atomically.

13. **Medium: declared LangGraph state does not match actual value types.** `FinAgentState.rag_query` is typed `Optional[str]` (`core/state.py:278`) but `retrieval_node` returns a `RAGQuery` object (`agents/retrieval.py:240-243`). This erodes the graph's typed-state contract and can break serializers/consumers. **Fix:** type it `Optional[RAGQuery]` (or return a string) and add state-handoff tests.

14. **Medium: tests are stale and do not verify the score path.** `tests/test_agent_pipeline.py:194-222` expects the old string query API and passes floats into `_rerank`, although `_rerank` now requires `ScoreBundle`; the checked-in suite fails one assertion and one error under `python -m unittest discover -s tests -v`. `pytest` is not installed in the venv. **Fix:** update tests to current `RAGQuery`/`ScoreBundle` contracts and add end-to-end assertions that final exposed scores are in [0,1].

15. **Low: KB document count is actually chunk count.** `backend/routers/knowledge_base.py:35-43` sets `total_documents` and `total_chunks` to the same `len(_documents)`, where `_documents` is the chunk list. The UI therefore overstates uploaded document count. **Fix:** count distinct `metadata['doc_name']` for documents.

16. **Low: the KB UI promises unsupported formats and parses results with the wrong shape.** It advertises/accepts TXT and MD (`frontend/src/pages/KnowledgeBase.tsx:98`, `114-119`) while API accepts only PDFs (`backend/routers/knowledge_base.py:53-55`); it renders `r.content`, `r.source`, and `r.score` (`156-162`) whereas API serializes `text`, `source_document`, and `similarity_score`. **Fix:** either implement text/Markdown ingestion or restrict the UI, and align the result DTO/rendering fields.

17. **Low: API-facing mutable Pydantic defaults are shared-looking and should be factories.** `backend/models/responses.py:18-22` declares list defaults as `[]`. Pydantic generally copies them, but this is brittle and easy to regress when code changes. **Fix:** use `Field(default_factory=list)` for all collection fields.

Chunking check: this is one of the stronger areas. Table regions are excluded from narrative extraction (`rag/ingest.py:178-213`), table chunks are labelled `chunk_type="table"` and split by row groups with repeating headers (`352-400`), so tables do not bleed into body chunks. There is no retrieval-time chunk-type policy, however, so numeric queries may still retrieve table chunks by normal relevance, which is expected unless a product rule says otherwise.

Structured-output check: both LLM call sites use Pydantic schemas via `.with_structured_output()` and propagate schema/invocation failures into explicit analysis error state or a labelled decision fallback (`agents/analysis.py:163-197`; `agents/decision.py:74-95`). No silent schema defaulting was found at those call sites.
